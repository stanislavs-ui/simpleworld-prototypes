## Когда показывается

Юзер на экране `sKhloeTask` (день N ≥ 3, любой из 3 типов: volume / buy / deposit). Тапает кнопку **«Проверить»** под основной CTA «Открыть Simple Wallet».

Вызов идёт на `POST /khloe/check` с `userId` и `khloe_current_day`. Бэк смотрит соответствующую метрику в Wallet (объём / покупка / депозит за период) и возвращает один из двух ответов:

- `{ status: "claimed" }` — задание выполнено, ритуал засчитан.
- `{ status: "not_yet" }` — задание ещё **не** выполнено (или Wallet не успел увидеть действие).

**Этот документ — про второй случай.**

---

## UI: что видит юзер

После 1.5s spinner'а («Проверяем...») — **остаётся на том же экране**. Никакой навигации, никаких bottom-sheet'ов, никаких отдельных экранов. Под кнопкой «Проверить» появляется **inline-блок**:

```
┌──────────────────────────────────────────────┐
│ ⓘ  Пока не засчитано. Бывает задержка —       │
│    попробуй ещё раз чуть позже.              │
└──────────────────────────────────────────────┘
```

**Стилистика блока:**
- Тёмный surface (`hsl(240 5% 8%)`, чуть темнее основной карточки)
- `border-left: 2px solid hsl(38 80% 50%)` (жёлтый/amber — нейтральный warn-цвет, не red)
- Иконка ⓘ (info-circle, 16px) `stroke="#eab308"`, выровнена по верху текста
- Текст: 13px, `color: hsl(0 0% 75%)`, line-height 1.5
- Появляется через `opacity: 0 → 1` (transition 0.3s ease) — без резких jump'ов

**Кнопка «Проверить» сразу возвращается в активное состояние.** Spinner исчезает. Юзер может тапнуть «Проверить» ещё раз — снова spinner, снова попытка.

---

## Финальный текст (Лира)

> **Пока не засчитано. Бывает задержка — попробуй ещё раз чуть позже.**

**Почему именно так:**
- «Пока не засчитано» — не «не выполнено» и не «провал». «Пока» подразумевает временность, не финальный приговор.
- «Бывает задержка» — снимает с юзера ощущение «я что-то сделал не так». Сразу даёт честное объяснение: иногда Wallet не успевает запушить событие.
- «Попробуй ещё раз чуть позже» — мягкое приглашение, не команда. Не «обнови страницу», не «попробуйте снова через X минут».
- Тон Хлои = спокойный mentor, не passive-aggressive, не обвинение.

---

## Поведение кнопки «Проверить»

- **Нет client-side cooldown.** Юзер может тапать «Проверить» сразу подряд сколько угодно раз.
- Каждый тап = новый `POST /khloe/check`, новый spinner 1.5s.
- При повторном тапе с уже видимым inline-сообщением — сообщение пропадает на время spinner'а, потом появляется снова (если `not_yet`) или начинается переход на `sKhloeRitualClaimed` (если `claimed`).
- Anti-spam — **на стороне бэка** (например, rate-limit 1 запрос в 30s, или дешёвая проверка через cached metric). Клиенту не дёргать таймер.

## Backend-контракт

### `POST /khloe/check`

**Request:**
```json
{ "userId": "u_123", "khloe_current_day": 5 }
```

**Response (выполнено):**
```json
{ "status": "claimed" }
```
→ Клиент: убрать spinner → `goToId('sKhloeRitualClaimed')` → одновременно вызвать `POST /khloe/claim` (или сервер автоматически уже отметил `khloe_day_5_claimed = true`).

**Response (ещё не выполнено):**
```json
{ "status": "not_yet" }
```
→ Клиент: убрать spinner → показать inline-блок «Пока не засчитано…» → кнопка «Проверить» возвращается.

**Опциональные поля в `not_yet`:**
- `progress` — текущий прогресс (если хотим показать «у тебя $14 из $20»). Сейчас в прототипе **не используем** — это отдельное продуктовое решение, и текст становится сложнее. Если решим показывать прогресс — добавим в следующий changelog.
- `last_seen_at` — последний момент, когда Wallet видел active session юзера (для дебага).

## Edge cases — что делать

| Кейс | Поведение |
|------|-----------|
| Юзер нажал «Проверить», получил `not_yet`, ушёл на карту, вернулся на Арену | `enterArena()` снова отдаёт `sKhloeTask` (день N не выполнен) — карточка с тем же типом, inline-блок **не показывается** (это новая сессия экрана) |
| Юзер закрыл вкладку и открыл заново | То же — inline-блок сбрасывается, юзер видит чистую карточку |
| Day flip случился прямо во время проверки (юзер задержался до полуночи) | `khloe_current_day` обновится на бэке. Если сервер видит, что юзер **сейчас** на day N+1 — возвращает `{ status: "not_yet" }` для day N. Клиент покажет inline. На следующий заход на Арену юзер увидит **новое** задание (для N+1). Старая `not_yet`-сессия не критична — просто чуть устаревшее состояние. |
| Сетевая ошибка / 5xx от `/khloe/check` | На клиенте — fallback inline-блок с **другим текстом** (не «бывает задержка», а «Что-то пошло не так. Попробуй ещё раз.»). В прототипе пока не реализован — добавим после реального бэка. |
| Юзер переключился на другую вкладку, вернулся через час | То же что «закрыл и открыл» — UI чистый. Если хочется auto-poll — это отдельная фича, в текущем changelog'е не нужно. |

## Демо в прототипе (для тестирования)

В sidebar секция «Демо-утилиты» добавлена кнопка:

> **🔁 Имитировать «не выполнено»**

Поведение:
- Тап → ставит флаг `localStorage.sw-khloe-task-fail-mock = "1"`. Alert: «Имитация „не выполнено" ВКЛ. Следующее «Проверить» = inline-сообщение про задержку.»
- Пока флаг включён, **любая** проверка (`checkKhloeTask`) возвращает failure UI.
- Повторный тап → флаг убирается. Alert: «Имитация „не выполнено" ВЫКЛ. Следующее «Проверить» = успех.»
- `resetAllQuests()` чистит флаг.

Этот toggle — **только для прототипа.** В проде его нет.

## Rationale: почему inline, а не отдельный экран

**Inline (выбранный подход):**
- ✅ Юзер не теряет контекст. Карточка с заданием, шаги, кнопка «Открыть Simple Wallet» — всё на месте.
- ✅ Психологически легче: «не получилось — но я уже здесь, тапну ещё раз».
- ✅ Меньше навигации = меньше fatigue.
- ❌ Inline-блок занимает место под кнопкой (но карточка и так длинная — 1 экран не теряется).

**Отдельный экран (отвергнутый):**
- ❌ Дополнительный навигационный шаг, юзер теряет фокус.
- ❌ Нужен ещё один экран в HTML, ещё один тип роутинга в `enterArena()`.
- ❌ После «Назад» юзер всё равно возвращается на тот же sKhloeTask — зачем разводить.
- ✅ Чуть более «премиум» feel (но недостаточно, чтобы перевесить minus'ы).

Inline — стандартный паттерн «inline-feedback» (как в форме при невалидном поле). Юзеру привычно.
