## Контекст

После онбординга Хлои (sD10e/sD10f, см. предыдущий changelog) юзер попадает на Home и видит на карте мира активный пин **Арена**. На следующий день он туда заходит — это **день 2 цикла Хлои**, первый «настоящий» заход.

День 2 — **подарочный**: Хлоя ничего не требует, но прежде чем подарить ритуал — отправляет юзера к Алексу выполнить социальное задание. Это сделано чтобы:
- удержать связку Алекс↔Хлоя (Алекс — главный, Хлоя — его «помощница»);
- заранее показать юзеру, что Арена и Хаб не изолированы — задания перекрёстные;
- мягко ввести в ежедневный ритуал, не давая полноценное задание сразу.

Реальные задания от Хлои начнутся с **дня 3** — по плану три типа заданий, чередующихся по кругу. Они будут описаны отдельным changelog после согласования дизайна.

## Backend / production-логика

### 1. Триггер «следующий день»

**В прототипе:** на Home добавлена demo-кнопка **«→ Следующий день»**, которая выставляет флаг `localStorage["sw-khloe-next-day"] = "1"`. Это симуляция flip-а дня.

**В production:** триггер day-2 — это **смена календарного дня (UTC 00:00)** относительно времени, когда юзер закончил онбординг Хлои (`sw-khloe-met = 1` от sD10f). Конкретнее:
- `khloe_onboarded_at` (timestamp выполнения sD10f) → серверная отметка.
- На следующий UTC-день (или по ТЗ — через N часов после онбординга) — у юзера активируется «day 2 Хлои».
- В state юзера должен быть параметр типа `khloe_current_day = 2` (с инкрементом каждый UTC-день).

**Почему важно:** без явного триггера юзер сразу после онбординга на Арене получит уже day-2 экраны — а должен сначала увидеть `sArena` (повтор правил), и только на следующий день — full день 2 цикл.

---

### 2. Социальное задание Алекса в хабе

**В прототипе:** на экране `sKhloeAlexTask` Алекс выдаёт стандартное соц-задание («Лайкнуть пост» → +10 к Ауре). Проверка — мок (1.5s спиннер, всегда успех).

**В production:**
- На день 2 у юзера в **хабе Алекса** должно быть **активное социальное задание** (любого из существующих типов: лайк, репост, подписка, проверка-комментария — то, что уже работает в серии Алекса).
- Хлоя при заходе на Арену **проверяет** — есть ли у юзера невыполненное активное соц-задание Алекса. Если есть → отправляет в хаб (sKhloeLocked). Если нет (или уже выполнено) → переходит к выдаче ритуала (sKhloeDay2).
- Кто кладёт задание в хаб — продуктовое решение: либо у юзера всегда висит «следующее в очереди» соц-задание Алекса, либо специально создаётся на flip дня. **Этот вопрос требует решения с продуктом** — пометьте, пожалуйста, как блокер.
- Награда за соц-задание — **только +10 к Ауре** (Алекс), ритуал даёт уже Хлоя на следующем экране.

---

### 3. Выдача ритуала на день 2

**В прототипе:** функция `completeKhloeDay2()` инкрементит `localStorage["sw-rituals-done"]` на +1, ставит флаг `sw-khloe-day2-claimed = "1"`, переходит на Home.

**В production:**
- За завершение дня 2 у Хлои (нажатие «Спасибо» на sKhloeDay2) — **+1 ритуал** в счётчик пользователя.
- Серверный флаг `khloe_day2_claimed = true` — повторная выдача ритуала на том же дне запрещена.
- На Home счётчик ритуалов растёт на 1.
- Контекст: к этому моменту у юзера в счётчике уже **10 ритуалов** (за серию Миры D1–D10) + **1 за день 2 Хлои = 11**. С дня 3 каждый выполненный день Хлои = +1 ритуал.

## Логика роутера `enterArena()`

Когда юзер на карте мира тапает пин **Арена** (после онбординга Хлои) — вызывается функция `enterArena()`. Логика:

```js
function enterArena() {
  // Сегодня ещё не «следующий день» → просто повтор правил.
  if (sw-khloe-next-day !== '1') {
    goToId('sArena');  // Экран 80 (повтор правил с онбординга)
    return;
  }

  // Day 2 активен. Алекс не помог + ритуал не выдан → блок.
  if (sw-khloe-alex-done !== '1' && sw-khloe-day2-claimed !== '1') {
    goToId('sKhloeLocked');  // Экран 81
    return;
  }

  // Алекс помог + ритуал не выдан → выдаём.
  if (sw-khloe-alex-done === '1' && sw-khloe-day2-claimed !== '1') {
    goToId('sKhloeDay2');  // Экран 83
    return;
  }

  // Ритуал уже выдан (повторный заход на той же дате) → повтор правил.
  goToId('sArena');
}
```

**Для production:** аналогичный роутер на бэке/фронте, но вместо `localStorage` — поля state юзера:
- `khloe_current_day` (2, 3, 4, ...)
- `khloe_alex_task_done` (boolean, сбрасывается на flip дня)
- `khloe_day_N_claimed` (boolean per day)

На последующих днях (3+) этот же роутер расширится: вместо sKhloeDay2 будет выбираться один из трёх типов заданий по кругу. Будет описано отдельно.
