## Backend / production-логика

### 1. Триггер дня

**В прототипе:** demo-кнопка «→ Следующий день» на Home инкрементит `localStorage["sw-khloe-current-day"]` на +1.

**В production:** триггер дня = смена UTC-дня (или интервал по ТЗ) с момента онбординга Хлои (`sw-khloe-met = 1`). На бэке у юзера хранится:
- `khloe_current_day` — целое (2, 3, 4, …), инкрементится 1 раз в сутки;
- `khloe_day_N_claimed` — boolean per day (или один JSON-массив), отмечает, что ритуал за день N выдан.

---

### 2. Цикл типов заданий

Реализовано **3 типа** заданий, **чередующихся по кругу**, начиная с дня 3:

| День N | Индекс цикла `(N − 3) mod 3` | Тип | Сумма |
|--------|------------------------------|-----|-------|
| 3 | 0 | **Объём торгов** | $20 |
| 4 | 1 | **Покупка криптовалюты** | $10 |
| 5 | 2 | **Пополнение баланса** | $10 |
| 6 | 0 | Объём | $20 |
| 7 | 1 | Покупка | $10 |
| 8 | 2 | Пополнение | $10 |
| … | … | … | … |

**Не рандом, а строгая последовательность.** Причина: рандом может выдать два дня подряд один тип — это плохой UX. Чередование гарантирует разнообразие и тренирует юзера на 3 разных use-case Wallet.

В прототипе:
```js
var khloeTaskCycle = ['volume', 'buy', 'deposit'];
function getKhloeTaskType(day) {
  return khloeTaskCycle[((day - 3) % 3 + 3) % 3];
}
```

### 4. Универсальное действие: «Открыть Simple Wallet»

Шаг 1 у всех трёх типов одинаковый: **«Нажми открыть Simple Wallet»**. Кнопка-CTA «Открыть Simple Wallet» под карточкой ведёт юзера прямо в нужный раздел приложения — куда конкретно, зависит от типа:

| Тип | Deep-link target |
|-----|------------------|
| Объём | Раздел **Trade** в Wallet |
| Покупка | Раздел **Buy** в Wallet |
| Пополнение | Раздел **Wallet → Deposit** |

**Зачем это важно:** в инструкции мы не пишем «зайди в раздел X» — потому что deeplink делает это за юзера. Это снимает когнитивную нагрузку и снижает количество поддержки. Антон, нужны 3 разных deep-link'а — уточни у Wallet команды (Владимир Барановский — продакт Simple Wallet, контакт через Стаса).

### 5. Сжигание ритуала (важно для логики)

**Что происходит, если юзер не выполнил задание за день:**
- Флаг `khloe_day_N_claimed` остаётся **false**.
- Счётчик ритуалов **не инкрементится**.
- На следующий день (N+1) юзер видит **новое** задание (по циклу).
- **Никакого отдельного экрана-уведомления «Ритуал сгорел»** мы не показываем — это решение продукта (29.04). Ритуал «тихо» сгорает, юзер видит только что Хлоя выдаёт следующее.

**Why так:** не хотим напоминать юзеру про потерю — это negative reinforcement и риск churn. Хотим, чтобы он просто двигался дальше.

**Серверная логика:** на flip дня бэк просто обновляет `khloe_current_day = N+1` и не трогает старый `khloe_day_N_claimed` — он остаётся `false` навсегда. Если в будущем нужна аналитика «сколько ритуалов сгорело у юзера», считается так: `count(khloe_day_N_claimed = false WHERE day < current_day AND day >= 3)`.

### 6. Проверка задания: «Пока не засчитано» (inline failure)

Если юзер нажал «Проверить» **до** того, как Wallet увидел его действие — мы **не уводим его на отдельный экран**, а показываем **inline-сообщение под кнопкой «Проверить»**:

> **«Пока не засчитано. Бывает задержка — попробуй ещё раз чуть позже.»**

Визуально: жёлтый ⓘ + светлый текст в плашке с жёлтым `border-left`. Появляется через `opacity 0 → 1` после 1.5s spinner'а.

**UX-правила:**
- Юзер остаётся на том же экране — никакой навигации, никаких bottom-sheet'ов.
- Кнопка «Проверить» **возвращается в активное состояние** сразу после показа сообщения. Юзер может тапнуть её ещё раз без cooldown'а на клиенте.
- При следующем «Проверить» — снова spinner, снова попытка проверить через `/khloe/check`.
- Без обвинений и passive-aggressive в копирайте: «бывает задержка» снимает ощущение «я что-то сделал не так» — задержка это нормально.

**Backend:**
- Endpoint `POST /khloe/check` возвращает один из:
  - `{ status: "claimed" }` → клиент переходит на `sKhloeRitualClaimed`, +1 ритуал
  - `{ status: "not_yet" }` → клиент показывает inline-сообщение
- **Анти-rate-limit на бэке** — например, 1 запрос в 30s (если нужен). На клиенте кнопка не блокируется.
- В прототипе `checkKhloeTask()` мокает оба state'а. Демо-toggle в sidebar: **«🔁 Имитировать «не выполнено»»** ставит флаг `sw-khloe-task-fail-mock`, и тогда любая проверка возвращает failure до тех пор пока флаг ВКЛ.

### 7. Повторный заход в тот же день после выполнения

Если юзер выполнил задание дня N и зашёл на Арену снова — он попадает на отдельный экран `sKhloeTaskDone`:

> Хлоя: «Сегодня ритуал засчитан. Приходи завтра — будет новое задание.»

Без CTA. Только back-стрелка на карту.

**Серверная логика:** при заходе на Арену проверяем `khloe_day_N_claimed === true` для текущего N → отдаём screen "task_done". Иначе → отдаём screen с задачей.

## Логика роутера `enterArena()` v2

```js
function enterArena() {
  // Без флага "следующий день" → повтор правил (старый sArena с онбординга)
  if (sw-khloe-next-day !== '1') return goToId('sArena');

  var day = sw-khloe-current-day || 2;

  // День 2 — подарочный (через соц-задание Алекса)
  if (day === 2) {
    if (sw-khloe-alex-done !== '1' && sw-khloe-day2-claimed !== '1') return goToId('sKhloeLocked');
    if (sw-khloe-alex-done === '1' && sw-khloe-day2-claimed !== '1') return goToId('sKhloeDay2');
    return goToId('sKhloeTaskDone');  // Повторный заход в тот же день
  }

  // День 3+ — реальное задание из цикла
  if (day >= 3) {
    if (sw-khloe-day-N-claimed === '1') return goToId('sKhloeTaskDone');
    var type = getKhloeTaskType(day);
    applyKhloeTaskMode(type);
    return goToId('sKhloeTask');
  }

  return goToId('sArena');  // Fallback
}
```

В **production**:
- Аналогичный роутер на бэке/фронте.
- Вместо `localStorage` — поля `khloe_current_day`, `khloe_alex_task_done`, `khloe_day_N_claimed` в state юзера.
- Перед рендером Арены — `GET /khloe/state?userId=` возвращает либо `task_today` (тип, сумма, шаги, статус задания), либо `task_done` (если уже выполнил), либо `gift_day_2` (если только-только перешёл на день 2 после онбординга).

## Финальная цепочка флоу день N (N ≥ 3)

```
Home (после онбординга)
   ↓ flip дня (UTC 00:00)  [в прототипе: demo-кнопка → инкремент khloe_current_day]
   ↓
Карта мира (s17) → пин «Арена» активен
   ↓ тап → enterArena()
   ↓
[если khloe_day_N_claimed === false]
   ↓
sKhloeTask (тип = TASK_CYCLE[(N-3) mod 3])
   ├─ юзер: тап «Открыть Simple Wallet» → deeplink → раздел Trade/Buy/Deposit
   ├─ юзер выполняет действие в Wallet ($20/$10/$10)
   └─ возвращается на экран → тап «Проверить»
       ↓ 1.5s spinner (опрос /khloe/check)
       ↓ выполнено → sKhloeRitualClaimed
                      ↓ «Спасибо» → +1 ритуал → Home
       ↓ не выполнено (в проде) → инлайн «Пока не вижу выполнения, попробуй ещё раз через пару минут»

[если khloe_day_N_claimed === true (повторный заход)]
   ↓
sKhloeTaskDone — «Сегодня ритуал засчитан. Приходи завтра.»
```
